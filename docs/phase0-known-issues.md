# VLA Phase 0 已知问题汇总（修订版）

> 编写：Pichai (PM) | 日期：2026-04-09 | 版本：v2

Phase 0 期间由 Pichai (PM)、Fabrice (Dev)、Moss (QA) 在执行过程中发现的全部已知问题。按严重程度分层，包含表现、根因、已有手段、缺失手段和当前状态。

## 🔴 阻塞性问题（需马上调整）

### 1. 测试链路信息丢失——PRD → 用例 → 执行 → 结论，每个交接点都可能丢信息

- **表现：** PRD 的验收标准到 Moss 手上变成模糊描述，expected result 缺失或降级，最终判定标准和 PRD 不一致。4/8 项目中 35 条测试只有 3 条用 mano-cua（应优先使用），说明指派时的执行要求也在传递中丢失
- **根因：** 制度层有规范但缺工具层自动校验，人工 enforce 有盲区——Moss 写用例时如果凭记忆而非回看 PRD 原文，PM 无法实时检测到
- **已有手段（pm-cockpit 制度层）：**
  - PRD 模板第 4 章 AC 编号 + 第 10 章验收索引 → 源头锚定
  - 七要素指派必须带 PRD 链接 → 传递锚定
  - Test Case 模板必须填 PRD 关联 + expected result 逐条展开 → 接收锚定
  - 测试指派附带执行规范地址 + 关键条款摘要
  - **作用：** 解决"不知道怎么做"，挡不住"知道怎么做但没做到位"
- **缺失手段（工具层 enforce）：**
  - 链路校验脚本：pre-translation（用例是否覆盖 PRD 全部验收点、expected result 是否为空）+ post-execution（报告是否遗漏验收点）
- **执行人：** Moss | **状态：🔶 制度层已落地，工具层待排期**

### 2. 执行产物质量无底线

- **表现：** 测试日志缺少观测摘要、expected result 为空也能出报告、observation 与 verdict 矛盾无人发现
- **根因：** 定义了"合格产出长什么样"（模板），但没有自动化质量门禁——不合格产物仍能走完流程
- **已有手段（pm-cockpit 制度层）：**
  - Test Report 模板第 6 章 mano-cua session 合规检查
  - 测试执行要求条款 9（expected result 逐条展开）、条款 11（evaluation 与日志交叉验证）
  - PM 收到 Test Report 后按模板逐项检查
  - **作用：** 定义了标准，PM 人工兜底能抓明显问题，但在几十条记录里肉眼找微妙降级不现实
- **缺失手段（工具层 enforce）：**
  - 产物质量门禁：expected result 覆盖率 < 100% → 阻断报告生成；observation 与 verdict 矛盾 → 自动标红；日志缺少 observation → 标记"需重跑"
- **执行人：** Moss | **状态：🔶 制度层已落地，工具层待排期**

### 3. 任务流转短路——开发直接交测试，跳过 PM

- **表现：** Fabrice 完成后直接 @Moss，PM 不知道交付物状态
- **根因：** 没有 enforce 的流转规则
- **方案：** pm-cockpit skill 明确"所有环节交付必须先回 PM"+ 七要素指派模板
- **状态：✅ 已落地**

### 4. 指令传达无 ACK 确认——Moss 停了 50 分钟无人察觉

- **表现：** 给 Moss "继续"指令后没确认收到，Moss 实际停了 50 分钟（3/31 事件）
- **根因：** 指令传达是单向发送，没有回执和验证机制
- **方案：** 三步确认：下达指令 → 等 ACK → 5 分钟后验证执行
- **状态：✅ 已落地**

### 5. 视觉呈现不达标——交付物看起来像毛坯房

- **表现：** md-wechat-formatter 功能完整但视觉远不如旧版 golden 版本，林菡因此被批评
- **根因：** 团队把视觉当"有空再美化"而非交付标准的一部分，开发和验收环节都没有视觉质量的卡点
- **方案：** 视觉呈现作为交付标准——有参照物先看再动手，没有参照物找同类作基线，交付前对比第一眼观感
- **状态：✅ 已写入长期规则，待 Phase 1 实操验证 enforce 效果**

## 🟡 需要短期解决（影响效率但不阻塞）

### 6. PM 进新群技能丢失——进了群不启动项目管理

- **表现：** 4/8 md-wechat-formatter 项目，PM 进群后 pm-cockpit 完全没启用，没有角色分工、没有启动仪式
- **根因：** GROUP.md 写入被埋在 Phase 1 第 6 步，中途断掉就是空的，新 session 读不到 pm-cockpit 标记
- **方案：** 收到"用 pm-cockpit"瞬间立刻写入 GROUP.md 基础信息（Step 0），后续补全
- **状态：✅ 已更新到 skill**

### 7. 巡检机制不自动启动

- **表现：** pm-cockpit-patrol cron 需要手动创建，忘了就没巡检
- **根因：** 巡检不是 skill 启动的标配动作
- **方案：** 启用 pm-cockpit 时自动创建巡检 cron（every 20m / timeout 300s）
- **状态：✅ 已更新到 skill**

### 8. 角色权限边界不清——智子越权执行

- **表现：** 智子（非团队成员）在 Phase 0 执行了不属于她的操作
- **根因：** 谁能做什么没有明确定义
- **方案：** 角色权限矩阵，由 PM 定义和 enforce
- **状态：✅ 已完成，推送到 pm-cockpit repo**

### 9. 跨 session 上下文断裂

- **表现：** session 重启后工作记忆清零，接不上之前做到哪了
- **根因：** 没有断点快照机制
- **方案：** 上下文快照模板，每次可能跨 session 时写入 memory
- **状态：✅ 模板已完成**
