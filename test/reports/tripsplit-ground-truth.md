## Buggy App 标注（负样本）

### B1 — TC-L1.1 创建旅行（数据持久化失败）

| 字段 | 值 |
|------|------|
| Session ID | `sess-20260401163408-53b524787daf4c88926c5cf2429f8e20`（STOPPED_BY_USER — 日期输入卡住后终止，未完成完整创建流程） |
| 补充验证 | curl POST /trip/create 确认数据未持久化 |
| 用例 | TC-L1.1 创建旅行 |
| 标注 | **incorrect** |
| 错误类型 | 数据丢失 (Data Loss) |
| 对应 Bug | B1 |
| 诊断描述 | `create_trip()` 函数执行 `db.execute(INSERT ...)` 后缺少 `db.commit()`。Flask 每个请求使用独立的数据库连接，POST 请求中的 INSERT 操作在未 commit 的情况下，连接关闭时事务自动回滚。后续 GET 请求的新连接无法看到未提交的数据，导致创建的旅行在页面重定向后立即消失。 |
| Golden 对照 | G1: 创建后刷新，旅行持久存在（Golden 版有 `db.commit()`） |
| 代码定位 | `buggy/app.py` L151: 注释 `# BUG: 漏了 db.commit()` |
| 轨迹完整性 | mano-cua 轨迹不完整（40 步后因日期输入困难终止），Bug 验证通过 curl 补充完成 |

### B2 — TC-L1.4 账单列表排序（排序方向反）

| 字段 | 值 |
|------|------|
| Session ID | `sess-20260401163705-fec48b3a997f4b5493ffbf9bee47fe7f` |
| 用例 | TC-L1.4 账单列表展示 |
| 标注 | **incorrect** |
| 错误类型 | 排序错误 (Sort Order Error) |
| 对应 Bug | B2 |
| 诊断描述 | 账单明细页的 SQL 查询使用 `ORDER BY date ASC, id ASC`，导致记录按日期正序排列（最早的在上，最新的在下）。PRD 要求按日期倒序展示（最新消费在最前面）。实际页面显示：餐饮4/1→交通4/2→住宿4/3（正序），应为住宿4/3→交通4/2→餐饮4/1（倒序）。 |
| Golden 对照 | G4: 按 DESC 倒序排列，最新记录在上 |
| 代码定位 | `buggy/app.py` L244/L249: `ASC` 应改为 `DESC` |
| 采集证据 | mano-cua 截图明确显示三条记录顺序为 4/1→4/2→4/3（正序） |

### B3 — TC-L1.5/L1.6 结算差额（计算方向反）

| 字段 | 值 |
|------|------|
| Session ID | `sess-20260401163824-6f338a615b094c79ae6ba5495004f5dd` |
| 用例 | TC-L1.5 差额计算 + TC-L1.6 结算方案 |
| 标注 | **incorrect** |
| 错误类型 | 逻辑错误 (Logic Error) |
| 对应 Bug | B3 |
| 诊断描述 | 结算差额计算公式写反：`balance = owed - paid`（应承担 - 实付），正确应为 `balance = paid - owed`（实付 - 应承担）。导致所有人的应收/应付方向完全反转。小明实付¥300应承担¥350，正确差额为-¥50（应付），Buggy 显示+¥50（应收）。结算方案中转账方向也随之反转：应为小明→小李¥50，Buggy 显示小李→小明¥50。 |
| Golden 对照 | G3/G5: 差额方向正确，结算方案转账方向正确 |
| 代码定位 | `buggy/app.py` L375-376: `owed[m] - paid[m]` 应改为 `paid[m] - owed[m]` |
| 采集证据 | mano-cua 截图显示小明+¥50/小红+¥200/小李-¥250，三人差额全部反转 |
